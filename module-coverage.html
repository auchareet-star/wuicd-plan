2Track B — โมดูลที่ “ไม่มีฐาน” (มหิดล Phase 2)

ต้องตัดสินใจแนวทางทุกตัว — ค่าเริ่มต้นที่แนะนำคือ Build ใหม่ เพราะรอ Mahidol Phase 2 (ยังไม่วางแผน) เสี่ยงเกินไปกับ deadline ที่ตายตัว

3Track A — โมดูลที่ “มีฐาน” Phase 1 (Fit/Gap ได้)

โมดูลที่ (สันนิษฐานว่า) อยู่ในมหิดล Phase 1 → ใช้แนวทาง Fit/Gap ตาม baseline 11 พ.ย. 2026 ได้

โมดูล (TOR 6.2.1)โมดูล (ต่อ)
(1) เวชระเบียน(14) ศูนย์สิทธิประโยชน์
(2) Patient bank + ติดตามการรักษา(15) ศูนย์ส่งต่อผู้ป่วย
(3) ห้องจ่ายยา/ผู้ป่วยนอก(17) คลังสินค้า/พัสดุ (Inventory)
(4) คลินิกทั่วไป(18) บัญชีลูกหนี้
(5) การเงินผู้ป่วยนอก(21) จัดการเรียกคิว
(6) ค่าตอบแทนแพทย์ (DF)(23) Data Interface (เชื่อมระบบ)
(7) รังสีวิทยา(24) บริหารบุคลากร
(11) นัดหมาย(26) IAM (ตัวตน/สิทธิ์เข้าถึง)
(12) จัดตารางปฏิบัติงานแพทย์(27) ขออนุมัติ (Request-Approve)
(13) เวชระเบียนอิเล็กทรอนิกส์/ภาพ(28) ขอความยินยอม (Consent)
⚠️ ต้องยืนยัน: รายการ Track A นี้เป็นการ สันนิษฐาน (เพราะไม่อยู่ในรายการ “ทำไม่ทัน” ของคุณ) — ต้องขอ “รายการโมดูลที่อยู่ใน Phase 1 baseline จริง” จากทีมมหิดลเป็นลายลักษณ์ เพราะบางตัวอาจยังไม่เสร็จเหมือนกัน

4กลยุทธ์: แบ่ง scope เป็น 2 Track

Trackแนวทางรายละเอียด
Track AFit/Gap บนฐาน Phase 1ทำตามแผนปรับ Dependency (Paper Fit/Gap → System Fit/Gap หลัง baseline 11 พ.ย.) — ดู แผนปรับ
Track BBuild ใหม่ (greenfield)ออกแบบ+พัฒนาเองตั้งแต่ต้น · เริ่มออกแบบได้เลยไม่ต้องรอมหิดล (เพราะไม่มีฐานอยู่แล้ว) · แต่ต้องการกำลัง Dev/Mobile เพิ่ม
ข้อสังเกตเชิงบวกของ Track B: เพราะ “ไม่มีฐาน” อยู่แล้ว จึง ไม่ติด dependency กับ baseline มหิดล — ออกแบบ/เริ่ม build ได้ทันทีคู่ขนานกับ Track A · ปัญหาคือ “กำลังคน + เวลา” ไม่ใช่ “รอของ”

5ตัวเลือกที่ต้องเคาะกับวลัยลักษณ์ (Track B)

ตัวเลือก 1 — Build เองทั้งหมดในกรอบเวลาเดิม

ต้อง เพิ่มกำลัง Dev + คนทำ Mobile อย่างมีนัยสำคัญ · เสี่ยงคุณภาพ/เวลาเพราะทีมไม่รู้ฐานเดิม · เหมาะถ้าเพิ่มทรัพยากรได้จริง

ตัวเลือก 2 — เจรจาส่งมอบเป็นเฟส (Phased delivery)

ส่ง Track A + Dental Lab + Mobile (หัวใจ) ก่อนตามกำหนด · โมดูลรอง (Micro/Patho/CSSD/IC/KPI/PR/MIS) เลื่อนเป็นเฟสถัดไป · ต้องแก้ไข/ตกลงในสัญญา

ตัวเลือก 3 — ผูกกับ Mahidol Phase 2

ขอให้มหิดลวางแผน Phase 2 ให้เสร็จทันวลัยลักษณ์ · เสี่ยงสูงสุด เพราะ Phase 2 ยังไม่วางแผน + มหิดลยังติด Phase 1 — ไม่แนะนำให้พึ่งเป็นทางหลัก
คำแนะนำ: ผสม ตัวเลือก 1 (สำหรับหัวใจ: Dental Lab + Mobile) + ตัวเลือก 2 (เจรจาเฟสสำหรับโมดูลรอง) — และยกเรื่องนี้เข้า escalation memo คู่กับเรื่อง baseline date

6สิ่งที่ต้องทำต่อทันที

  1. ขอรายการ “Phase 1 baseline modules” จริงจากมหิดล เป็นลายลักษณ์ → ยืนยัน Track A/B ให้แน่ (ตอนนี้ Track A ยังเป็นการสันนิษฐาน)
  2. ทาน TOR ต้นฉบับ ว่า “ผ่าตัด+วิสัญญี” และ “หอผู้ป่วยใน” อยู่ในขอบเขตวลัยลักษณ์หรือไม่
  3. เพิ่มประเด็น Track B เข้า escalation memo (คู่กับ baseline date) — แจ้งว่าส่วนนี้ไม่มีฐาน ต้อง build ใหม่ + ขอตัดสินใจทรัพยากร/เฟส
  4. ประเมิน effort Track B (โดยเฉพาะ Dental Lab + Mobile 3 แอป) เพื่อรู้ว่าต้องเพิ่มคนกี่คน
  5. เริ่มออกแบบ Track B คู่ขนานได้เลย (ไม่ต้องรอ baseline) — ใช้ req วลัยลักษณ์ตรงๆ
เอกสารวิเคราะห์ scope ภายใน · รายการ Track A เป็นการสันนิษฐาน—ต้องยืนยันกับมหิดล · ทานรายการโมดูล + ขอบเขตกับ TOR ต้นฉบับ (ไฟล์สแกน) ก่อนใช้จริง
