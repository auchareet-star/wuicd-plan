3แผนวลัยลักษณ์ที่ปรับให้สมจริง — แยก Fit/Gap เป็น 2 ระยะ

ระยะช่วงเวลาทำอะไร / ปรับจากเดิมอย่างไร
Phase 0 Setupมิ.ย.–ก.ค.เดิม + เพิ่ม: ขอ artifact มหิดล (req spec, เคส UAT, SAP IAM, DB schema, source) เป็นฐานความรู้แทนการพึ่งคน
Phase 1a Paper Fit/Gap ✅ก.ค.–ต.ค.ใหม่: เทียบ req วลัยลักษณ์ ↔ req มหิดล (ที่ระบบถูกสร้างมา) — ทำได้เลยไม่ต้องรอระบบ · domain มาจากผู้ใช้วลัยลักษณ์
Phase 2 Design (M2)ก.ย.–8 พ.ย.ออกแบบจาก Paper Fit/Gap · ติดป้าย “Provisional — รอยืนยันกับ baseline” ในเอกสารส่งมอบ M2
— GATE: baseline มหิดลพร้อม ~11 พ.ย. 2026 —
Phase 1b System Fit/Gap ⚠️กลาง พ.ย.–ธ.ค.ใหม่: ยืนยัน gap ที่วิเคราะห์บนกระดาษ กับระบบจริง → ปรับ design · คาบกับ Build ตอนต้น (เผื่อ rework)
Phase 3 Buildพ.ย.–ก.พ.Config + Develop gap + Integration/Migration โดยเรียนจากระบบ Phase 1 ที่นิ่ง + เอกสาร (ทีมมหิดลไม่ว่าง—ทำ Phase 2 ต่อ) · + Track B build โมดูลใหม่คู่ขนาน (ดู Module Coverage)
Phase 4–5 Test/Go-Liveม.ค.–7 เม.ย.เหมือนเดิม แต่ buffer หดเพราะ Phase 1b กินเวลา → ต้องเฝ้าระวัง
หลักการ: ดึงทุกอย่างที่ “ไม่ต้องใช้ระบบมหิดล” มาทำก่อน (req วลัยลักษณ์, Paper Fit/Gap, To-be design) เพื่อให้ตอน baseline มา เหลือแค่ “ยืนยัน + ต่อระบบ” ซึ่งเร็ว — ลดเวลาที่เป็นคอขวด

4Risk Register (ปรับใหม่ตามความจริง)

ความเสี่ยงระดับการรับมือ
R1 มหิดลเลื่อนจาก 11 พ.ย. (ตอนนี้ติด UAT)สูงมากขอ commit baseline date เป็นลายลักษณ์, ทำ Paper Fit/Gap คู่ขนาน, กำหนด drop-dead date, แจ้งลูกค้าเร็ว (ดู memo ข้อ 6)
R2 ฐานเปลี่ยนระหว่างทาง (ยังพัฒนาอยู่)สูงFreeze เวอร์ชันที่จะ Fit/Gap, จัดการผ่าน Change Request, ออกแบบ M2 เป็น provisional
R3 ทีมมหิดลไม่ว่างตลอด (Phase 1 → ต่อ Phase 2) + ไม่มีใครรู้ระบบสูงมากพึ่งทีมไม่ได้ → เรียนจาก ระบบ Phase 1 ที่นิ่ง + เอกสาร + เคส UAT, ขอ async Q&A เท่าที่ได้, ให้ Dev+BA reverse-engineer เป็นงานหลัก
R4 ทีมเรา 7 คน ไม่ครบบทบาทตาม TORกลาง-สูงเคาะคนทำ Migration/QA/Integration, จัดคนเพิ่มจุดวิกฤต หรือ re-scope, เคลียร์ compliance กับวลัยลักษณ์ (ดูข้อ 5)

พร้อมส่ง — ปรับชื่อ/วันที่แล้วใช้ได้เลย · เป้าหมาย: ล็อก baseline date + ให้ลูกค้ารับทราบว่า M2 จะเป็น Paper Fit/Gap

เรื่อง Dependency ของระบบ DIS วลัยลักษณ์ ต่อ baseline ของ HIS ฐานมหิดล และผลต่อกำหนดส่งงวด M2

1. บริบท — ระบบ DIS ของวลัยลักษณ์พัฒนาต่อยอดจาก HIS ฐานมหิดล ซึ่งปัจจุบันยังอยู่ระหว่าง UAT/Pilot และมีกำหนด Go-live ระลอกสุดท้าย (baseline) ประมาณ 11 พ.ย. 2026

2. ข้อเท็จจริงด้านเวลา

  • กำหนดส่งงวด M2 (Fit/Gap + ออกแบบ) ของวลัยลักษณ์ = 8 พ.ย. 2026
  • baseline ฐานมหิดลพร้อมใช้ = ~11 พ.ย. 2026 (หลัง M2)

3. ผลกระทบ — ไม่สามารถทำ Fit/Gap กับ “ระบบจริงที่นิ่งแล้ว” ให้เสร็จทันงวด M2 ได้ ดังนั้นเอกสารงวด M2 จะจัดทำบนฐาน “Paper Fit/Gap” (เทียบความต้องการวลัยลักษณ์กับข้อกำหนดที่ระบบมหิดลถูกสร้างมา) และถือเป็น ฉบับชั่วคราว (provisional) ที่ต้องยืนยันกับ baseline จริงในช่วง Build

4. ความเสี่ยง — หากฐานมหิดลเลื่อนจาก 11 พ.ย. 2026 จะกระทบ Build, การเชื่อมต่อ, การโอนย้ายข้อมูล และอาจทำให้ Go-Live (7 เม.ย. 2027) เลื่อน รวมถึงประเด็นค่าปรับตามสัญญา

5. สิ่งที่ขอการยืนยัน/สนับสนุน

  1. ยืนยัน baseline date ของฐานมหิดลเป็นลายลักษณ์อักษร และแจ้งทันทีหากมีการเลื่อน
  2. รับทราบและยอมรับแนวทาง Fit/Gap แบบ 2 ระยะ (Paper ก่อน → System หลัง baseline) สำหรับงวด M2
  3. เนื่องจากทีมมหิดลจะทำ Phase 2 ต่อ (ไม่ว่าง) ขออย่างน้อย ช่องทาง async Q&A + สิทธิ์เข้าถึงระบบ Phase 1 ที่นิ่ง + เอกสาร เพื่อให้ทีมเราศึกษา/เชื่อมต่อเองได้
  4. ขอบเขต Track B (โมดูลไม่มีฐาน: Dental Lab, Mobile, Micro/Patho, MIS ฯลฯ): ต้อง build ใหม่ — ตกลงแนวทาง (เพิ่มคน / ส่งเป็นเฟส) ดู Module Coverage
  5. ส่งมอบ เอกสาร req / เคส UAT / แนวทาง SAP Interface (IAM) / DB schema ของฐานมหิดล เป็นฐานความรู้
  6. ตัดสินใจเรื่อง กำลังคนบทบาทที่ยังขาด (Data Migration, QA, Integration) — จัดเพิ่มหรือใช้ทีมมหิดลร่วม

6. Drop-dead date — เพื่อรักษากำหนด Go-Live 7 เม.ย. 2027 ฐานมหิดลควรนิ่ง (baseline) ไม่เกิน [ระบุวัน เช่น ปลาย พ.ย. 2026] หากเกินกว่านี้ ขอเสนอให้ทบทวนกำหนดการ/ขอบเขตงวดร่วมกัน
