3กำลังคนที่ต้องใช้ — แยกตามช่วงเวลา

Phase 0 · Setup

มิ.ย. 2026
นำ: PM + System Engineer
ทำ: kick-off, แผน 9 ฉบับ, ตั้ง environment, จองคิวทีม HIS เดิม · โหลดต่ำ (~30%) — โอกาสเตรียมงานล่วงหน้า

Phase 1 · Discovery + Fit/Gap

ก.ค.–ต.ค. 2026
นำ: System Analyst + DB Analyst
ต้องมี: ทีม HIS เดิม (Sprint A–C) · Programmer ที่ยังว่างดึงมาช่วยทำ RTM/inventory ได้

Phase 2 · Design (เอกสาร M2)

ก.ย.–พ.ย. 2026
นำ: SA + UX + Graphic UI + DB Designer
ทับกับ Phase 1 → นี่คือเหตุที่ ก.ย.–ต.ค. เป็น peak

Phase 3 · Build

พ.ย. 2026 – ก.พ. 2027
นำ: Senior Programmer + Programmer×3 + System Engineer
ต้องมี: ทีม HIS เดิม (Sprint D) · เริ่มดึง QA เขียน test case คู่ขนาน

Phase 4 · Test/Train/Migrate

ม.ค.–มี.ค. 2027
นำ: QA + Sr Data Engineer + Project Coordinator (อบรม)
ทับกับ Phase 3 → เหตุที่ ก.พ.–มี.ค. เป็น peak · Dev ยังปิดงานขณะ QA เริ่มแล้ว

Phase 5 · Go-Live

มี.ค.–เม.ย. 2027
นำ: System Engineer + DBA + PM
ต้องมี: ทีม HIS เดิม standby ช่วง cutover · ทุกคน on-call ช่วง Hypercare

4จัดการ 2 ช่วง Peak (ข้อเสนอ)

🔴 Peak 1 — ก.ย.–ต.ค. 2026 (Fit/Gap ทับ Design)

  • ดัน RTM + inventory ระบบเดิมให้เริ่มตั้งแต่ ก.ค.–ส.ค. (ใช้ Programmer ที่ยังว่าง) เพื่อให้ Fit/Gap ไม่กระจุกปลายช่วง
  • แยก SA เป็น 2 หมวก: คนหนึ่งปิด Fit/Gap อีกคน (หรือ UX/DB Designer) เริ่ม Design คู่ขนาน
  • ล็อกคิวทีม HIS เดิม Sprint C (review Fit/Gap) ให้ตรงต้น ต.ค. ห้ามเลื่อน — เป็นคอขวด
  • ห้ามทีมหลัก (SA, DB Designer, UX) ลาพักร้อนช่วงนี้

🔴 Peak 2 — ก.พ.–มี.ค. 2027 (Build ทับ Test/Train)

  • เริ่ม SIT ทีละโมดูลที่เสร็จก่อน (ไม่รอ build ครบ) เพื่อเกลี่ยงาน QA
  • เตรียมสื่อ/คู่มืออบรมตั้งแต่ ม.ค. เพื่อให้ มี.ค. โฟกัสจัดอบรมอย่างเดียว
  • วางแผน OT / คนเสริมชั่วคราวเฉพาะ Programmer + QA 4–6 สัปดาห์นี้
  • Migration sync กับทีม HIS เดิม (ก.พ.) ต้องเสร็จก่อนเข้า rehearsal
หลักเกลี่ยโหลด: "ดึงงานที่ทำได้ล่วงหน้ามาทำตอนว่าง" — RTM/Test Case/UI prototype/สื่ออบรม ไม่ต้องรอให้ถึงเฟสของมัน เริ่มได้ทันทีที่มี input

ทรัพยากรคอขวด — ต้องล็อกปฏิทินทั้ง 6 ช่วงใน MOU ตั้งแต่ Day 0 และจับคู่คนฝั่งเราให้ชัด

6ประเด็นสำหรับที่ประชุมภายใน (ต้องเคาะ)

Q1 — ทีม HIS เดิม: ตกลงโควตา (กี่วัน-คน/เดือน) และ lead time การจองได้หรือยัง? ใครเป็น SPOC สองฝั่ง?
Q2 — Peak 2 ช่วง: จะรับมือ ก.ย.–ต.ค. และ ก.พ.–มี.ค. ด้วยวิธีไหน — OT / จ้างเสริมชั่วคราว / ห้ามลา / เกลี่ยงานล่วงหน้า?
Q3 — Programmer ช่วง มิ.ย.–ต.ค.: ยังไม่เข้าเฟส build จะให้ทำอะไร (ช่วย RTM / prototype / เขียน test case) หรือเข้าโครงการทีหลังเพื่อประหยัด?
Q4 — Bus factor: แต่ละบทบาทมี backup ไหม โดยเฉพาะ Sr Data Engineer และ System Engineer ที่อยู่บน critical path
Q5 — วันหยุดปีใหม่ (1 ม.ค.): ตกกลางเฟส Build — จะปรับแผน sprint อย่างไรไม่ให้เสีย momentum?
Q6 — Environment/เครื่องมือ: Dev/Test/repo/ALM พร้อมภายในเมื่อไร (ต้องเสร็จใน Phase 0)?

สิ่งที่ต้องตัดสินใจออกจากห้องประชุม

  • ยืนยันตารางจองทีม HIS เดิม 6 ช่วง + ผู้รับผิดชอบจับคู่
  • แผนรับมือ Peak 2 ช่วง (เลือกวิธี + งบ OT/คนเสริม ถ้ามี)
  • มอบหมายงาน Programmer ช่วงต้นโครงการ
  • กำหนด backup รายบทบาทบน critical path
  • วันเริ่ม onboarding แต่ละตำแหน่ง (ไม่ต้องเข้าพร้อมกันหมด)
