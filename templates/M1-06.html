1วัตถุประสงค์และขอบเขตการทดสอบ

1.1 วัตถุประสงค์

  • ยืนยันว่าระบบ DIS ทำงานได้ตรงตามข้อกำหนด (Requirements) และเอกสารออกแบบในงวด M2
  • ตรวจสอบการเชื่อมต่อ (Integration) กับ HAIT Plus, SAP, PACS, LIS/RIS, WU-HRMS ทำงานถูกต้องครบถ้วน
  • ลดความเสี่ยงของข้อบกพร่อง (Defect) ระดับ Critical/High ก่อนขึ้นใช้งานจริง (Go-Live งวด M4)

1.2 ขอบเขต (In-scope)

  • โมดูล HIS ตามภาคผนวก ก ส่วนที่ 1 (เวชระเบียน, คลินิกทันตกรรม, ห้องยา, LIS, PACS, Dental Lab, นัดหมาย, การเงิน/ลูกหนี้, พัสดุ, Consent)
  • ระบบการเรียนการสอน ตามภาคผนวก ก ส่วนที่ 2
  • Mobile Application 3 ตัว: Patient / Student / Dentist
  • จุดเชื่อมต่อทุก Interface และการ Migrate ข้อมูลจากระบบเดิม

1.3 นอกขอบเขต (Out-of-scope)

  • ระบุงานที่ไม่รวมในแผนทดสอบ เช่น โครงสร้างเครือข่ายของหน่วยงาน ฯลฯ

2ระดับการทดสอบและเกณฑ์เริ่ม/จบ (Test Levels & Entry/Exit Criteria)

ระดับขอบเขต/ผู้รับผิดชอบเกณฑ์เริ่ม (Entry)เกณฑ์จบ (Exit)
Unit Testฟังก์ชันย่อย / Programmerโค้ดผ่าน build, มี test caseผ่าน ≥ 80% coverage, ไม่มี bug บล็อก
SIT
(System Integration)		การเชื่อมต่อระหว่างโมดูล/ระบบภายนอก / QA + DevUnit ผ่าน, ติดตั้งบน Test env, Interface พร้อมทุก Interface ส่ง-รับข้อมูลถูกต้อง, ไม่มี Critical/High ค้าง
UAT
(User Acceptance)		ผู้ใช้จริงตามกระบวนงาน / Key-user + QASIT ผ่าน, มี UAT script, ข้อมูลทดสอบพร้อมผู้ใช้ลงนามรับรอง, ไม่มี Critical/High ค้าง
E2E
(End-to-End)		กระบวนงานเต็มเส้น (ผู้ป่วยเข้า→รักษา→เบิกจ่าย) / QAโมดูลที่เกี่ยวข้องผ่าน SITทุก business flow หลักผ่านครบ
Performanceโหลด/ความเร็ว/ความทนทาน / System Engineer + QAระบบเสถียรบน env ใกล้เคียง Productionผ่านเกณฑ์ตอบสนอง ≤ 3 วิ ที่ __ users พร้อมกัน
Securityช่องโหว่/สิทธิ์เข้าถึง/PDPA / Security + QAระบบติดตั้งครบ, มี checklist OWASPไม่พบช่องโหว่ระดับ High ขึ้นไป

SIT จะทดสอบกรณีส่งผลตรวจ LIS กลับเข้าเวชระเบียนทันตกรรมของผู้ป่วยให้แสดงผลถูกต้องภายในเวลาที่กำหนด

3นิยามความรุนแรงของข้อบกพร่อง (Defect Severity) และ SLA การแก้ไข

ระดับนิยามSLA แก้ไข (เริ่ม/แก้เสร็จ)ผลต่อ Go-Live
Criticalระบบหยุดทำงาน/ข้อมูลเสียหาย/กระทบความปลอดภัยผู้ป่วย — ไม่มีทางเลี่ยงเริ่มทันที / ภายใน 4 ชม.บล็อก ห้ามขึ้นระบบ
Highฟังก์ชันหลักใช้งานไม่ได้ มีทางเลี่ยงยากภายใน 1 วันทำการบล็อก
Mediumฟังก์ชันรองผิดพลาด มีทางเลี่ยงได้ภายใน 3 วันทำการไม่บล็อก (มีแผนแก้)
Lowข้อบกพร่องเล็กน้อย/ความสวยงาม/ข้อความภายใน งวดถัดไปไม่บล็อก
⚠️ เกณฑ์ Go-Live: ต้องไม่มี Defect ระดับ Critical และ High ค้างอยู่ จึงจะผ่าน Exit ของ UAT และอนุมัติขึ้นระบบจริง

4สภาพแวดล้อมและข้อมูลทดสอบ (Test Environment & Data)

4.1 สภาพแวดล้อม

สภาพแวดล้อมวัตถุประสงค์หมายเหตุ/สเปก
DEVพัฒนา + Unit Test________
SIT / TESTทดสอบเชื่อมต่อ + E2Eแยกจาก Production, มี mock ระบบภายนอก
UAT / Stagingผู้ใช้ทดสอบ + Performanceสเปกใกล้เคียง Production

4.2 ข้อมูลทดสอบ (Test Data)

  • ใช้ข้อมูลจำลอง (Synthetic) สำหรับเคสทั่วไป และข้อมูลจริงที่ปกปิด (Masked/De-identified) สำหรับ Migration ตาม PDPA
  • เตรียมชุดข้อมูลผู้ป่วยตัวอย่างครอบคลุมเคสทันตกรรม (ถอนฟัน, อุดฟัน, รักษารากฟัน, จัดฟัน, X-ray)
  • ผู้รับผิดชอบเตรียมข้อมูล: DB Analyst / DBA

เตรียมผู้ป่วยจำลอง 1 ราย ที่มีประวัติ X-ray (PACS) + ผลตรวจ LIS + ใบสั่งยา เพื่อทดสอบ flow ครบทั้งระบบ
