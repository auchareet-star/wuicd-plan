1ภาพรวมและบทบาทผู้ดูแลระบบ

1.1 ภาพรวมระบบ DIS — ระบบ DIS เป็นระบบสารสนเทศด้านทันตกรรมแบบ Web + Mobile (3 แอป) เชื่อมต่อกับ HIS เดิม (HAIT Plus) และระบบเชื่อมโยงอื่น ๆ ผู้ดูแลระบบมีหน้าที่ดูแลให้ระบบพร้อมใช้งานอย่างต่อเนื่อง

1.2 บทบาทและหน้าที่ของผู้ดูแลระบบ (Administrator)

บทบาทหน้าที่หลัก
ผู้ดูแลระบบ (System Admin)จัดการผู้ใช้/สิทธิ์, ดูแล Master Data, เฝ้าระวังระบบ, ประสานงาน Support
ผู้ดูแลฐานข้อมูล (DBA)สำรอง/กู้คืนข้อมูล, ดูแลประสิทธิภาพฐานข้อมูล, ตรวจสอบ log
ระบุชื่อ-สังกัด ผู้รับผิดชอบจริงระบุขอบเขตงาน

2การจัดการผู้ใช้และสิทธิ์ (User & Role Management)

2.1 ขั้นตอนเพิ่ม / ลบ / แก้ไขผู้ใช้

  1. เข้าเมนู ตั้งค่า ▸ จัดการผู้ใช้ ด้วยบัญชีที่มีสิทธิ์ Admin
  2. กด เพิ่มผู้ใช้ ▸ กรอกข้อมูล (ชื่อ, รหัสพนักงาน/นักศึกษา, อีเมล, หน่วยงาน)
  3. กำหนด role และผูกกับสิทธิ์ (Permission Group) ▸ บันทึก
  4. การลบ/ระงับใช้งาน: เลือกผู้ใช้ ▸ ระงับการใช้งาน (แนะนำให้ระงับแทนการลบ เพื่อคง Audit log)

2.2 ตารางสิทธิ์ตาม role (ตัวอย่าง — ปรับตามนโยบายจริง)

เมนู / ฟังก์ชันทันตแพทย์นักศึกษาเวชระเบียนเภสัชการเงินแอดมิน
เวชระเบียน/บันทึกการรักษาRWRW*RWR—R
สั่งยา/ห้องยาRWRW*—RW—R
การเงิน/ลูกหนี้R—R—RWR
จัดการผู้ใช้/ตั้งค่าระบบ—————RW

RW=อ่าน/เขียน, R=อ่านอย่างเดียว, —=ไม่มีสิทธิ์, * = นักศึกษาต้องผ่านการรับรอง (Co-sign) จากอาจารย์ทันตแพทย์

⚠️ หลักการสิทธิ์ขั้นต่ำ (Least Privilege): กำหนดสิทธิ์เท่าที่จำเป็นต่อหน้าที่ และทบทวนสิทธิ์ผู้ใช้อย่างน้อยทุก 6 เดือน

3.2 ขั้นตอนกู้คืน (Restore)

  1. แจ้ง/ขออนุมัติการกู้คืนตามขั้นตอน (ระบุผู้มีอำนาจอนุมัติ)
  2. เลือกชุดสำรองที่ต้องการ (ตรวจสอบวันที่/เวลา และความสมบูรณ์ของไฟล์)
  3. กู้คืนลงระบบสำรอง (Staging) ก่อนเพื่อทดสอบ แล้วจึงนำขึ้น Production
  4. ตรวจสอบความถูกต้องของข้อมูล (reconciliation) และบันทึกผลลงทะเบียน

5การจัดการ Master Data

Master Data คือข้อมูลตั้งต้นที่ใช้ร่วมกันทั้งระบบ การแก้ไขต้องระมัดระวังเพราะกระทบหลายโมดูล

บัญชียา (Drug Master)รหัสยา, ขนาด, ราคา, คลังเภสัชกร
สิทธิการรักษา/กองทุนเบิกได้, ประกันสังคม, เงินสดการเงิน/เวชระเบียน

⚠️ ก่อนแก้ไข Master Data: สำรองข้อมูลชุดนั้นก่อนเสมอ และทดสอบในระบบ Staging — การลบรหัสที่มีรายการอ้างอิงอยู่อาจทำให้รายงานย้อนหลังผิดพลาด แนะนำให้ใช้สถานะ “ยกเลิกการใช้งาน” แทนการลบ

6ตารางการแก้ปัญหาเบื้องต้น (Troubleshooting)

อาการสาเหตุที่เป็นไปได้วิธีแก้เบื้องต้น
ผู้ใช้ Login ไม่ได้รหัสผ่านหมดอายุ / บัญชีถูกระงับ / สิทธิ์ผิดรีเซ็ตรหัสผ่าน, ตรวจสถานะบัญชีและ role ในเมนูจัดการผู้ใช้
ระบบช้า / ค้างCPU/Memory สูง, Query หนัก, Disk เต็มตรวจ Monitoring, เคลียร์พื้นที่, รีสตาร์ทบริการตามขั้นตอน, แจ้ง DBA
ข้อมูลผู้ป่วยจาก HAIT Plus ไม่ขึ้นInterface ค้าง / queue ติด / token หมดอายุตรวจ Integration log, รัน re-sync, หากแก้ไม่ได้แจ้งทีม HIS เดิม (ดู M4-07)
พิมพ์เอกสาร/ใบเสร็จไม่ออกเครื่องพิมพ์ออฟไลน์ / ตั้งค่า template ผิดตรวจสถานะเครื่องพิมพ์และ driver, ตรวจการตั้งค่ารายงาน
รายงานการเงินไม่ตรงMaster ราคา/สิทธิ์ไม่อัปเดตตรวจ Master Data, ตรวจช่วงเวลาดึงรายงาน, แจ้งการเงิน
เพิ่มอาการที่พบบ่อย

7การ Update / Patch ระบบ

  1. รับแจ้ง Patch/Update จากผู้รับจ้าง พร้อม Release Note และรายการเปลี่ยนแปลง
  2. นัดหมายช่วงเวลานอกเวลาให้บริการ (Maintenance Window) และแจ้งผู้ใช้ล่วงหน้า
  3. สำรองข้อมูล + สำรองเวอร์ชันปัจจุบัน (Rollback point) ก่อนติดตั้ง
  4. ติดตั้งและทดสอบในระบบ Staging ก่อน แล้วจึงขึ้น Production
  5. ทดสอบฟังก์ชันหลัง Patch (Smoke Test) และจุดเชื่อม HAIT Plus
  6. บันทึกผลลงทะเบียนการเปลี่ยนแปลง (Change Log) และแจ้งปิดงาน
หมายเหตุ: การ Update/Patch ในช่วงประกัน 2 ปี อยู่ในขอบเขตบริการของผู้รับจ้างตามเอกสาร Warranty & SLA (M4-07)

8ความปลอดภัย (Security)

  • Audit log: ระบบบันทึกการเข้าถึง/แก้ไขข้อมูลสำคัญ (ใคร-ทำอะไร-เมื่อไร) ห้ามปิด และต้องเก็บอย่างน้อย 90 วัน
  • การจัดสิทธิ์: ยึดหลักสิทธิ์ขั้นต่ำ, แยกหน้าที่ (Segregation of Duties), ทบทวนสิทธิ์เป็นระยะ
  • รหัสผ่าน: กำหนดความซับซ้อน, อายุรหัสผ่าน, ล็อกบัญชีเมื่อใส่ผิดเกินกำหนด
  • ข้อมูลส่วนบุคคล (PDPA): เข้าถึงข้อมูลผู้ป่วยตามหน้าที่เท่านั้น, mask ข้อมูลที่ไม่จำเป็น
  • บัญชีผู้ดูแล: ใช้บัญชีเฉพาะบุคคล หลีกเลี่ยงบัญชีกลางร่วมกัน
