1ภาพรวมสถาปัตยกรรม (Architecture Overview)

ระบบ DIS ออกแบบเป็นสถาปัตยกรรมแบบหลายชั้น (Multi-tier / N-tier) แยกชั้นการนำเสนอ (Presentation), ชั้นแอปพลิเคชัน (Application/API), ชั้นข้อมูล (Data) และชั้นการเชื่อมต่อ (Integration) ออกจากกัน เพื่อรองรับการขยายและการบำรุงรักษา

┌──────────────── CLIENT TIER ─────────────────┐ │ Web App Mobile: Patient Student Dentist│ │ (Browser) (iOS/Android — 3 แอป) │ └───────────────────────┬──────────────────────┘ │ HTTPS / TLS 1.3 ┌──────▼───────┐ │ Load Balancer │ (WAF + Reverse Proxy) └──────┬───────┘ ┌───────────────────────▼──────────────────────┐ │ APPLICATION TIER │ │ API Gateway → App Server (x N instances) │ │ ┌─────────┬─────────┬─────────┬────────────┐ │ │ │ เวชระเบียน│ หัตถการ │ นัดหมาย │ การเงิน/ยา │ │ │ └─────────┴─────────┴─────────┴────────────┘ │ │ Auth/SSO · Cache (Redis) · Message Queue │ └──────┬─────────────────────────┬──────────────┘ │ │ ┌────────▼────────┐ ┌────────▼─────────────┐ │ DATA TIER │ │ INTEGRATION TIER │ │ DB (PostgreSQL / │ │ ESB / API Adapter │ │ SQL Server / │◄─────►│ ┌──────┬──────┬────┐ │ │ Oracle) │ │ │HAIT+ │ SAP │PACS│ │ │ + Object Storage │ │ ├──────┼──────┴────┤ │ │ (PACS images) │ │ │LIS/RIS│ HL7/FHIR │ │ └──────────────────┘ │ └──────┴───────────┘ │ └───────────────────────┘

รูปแบบการเชื่อมต่อ: HL7 v2 / FHIR สำหรับข้อมูลคลินิก, DICOM สำหรับภาพรังสี (PACS), Web Service/API สำหรับ SAP และ LIS/RIS

4สถาปัตยกรรมการเชื่อมต่อ (Integration Architecture)

ระบบปลายทางโพรโทคอล/มาตรฐานทิศทางข้อมูลที่แลกเปลี่ยน
HAIT Plus (HIS เดิม)HL7 v2 / FHIR / APIสองทางผู้ป่วย (HN), visit, สิทธิ์การรักษา, วินิจฉัย
SAPWeb Service / IDoc / APIสองทางการเงิน, พัสดุ/คงคลังยา, บัญชี
PACSDICOM / WADOสองทางภาพรังสีทันตกรรม (X-ray/CBCT)
LIS / RISHL7 v2 / APIสองทางคำสั่งตรวจ, ผลแลป, ผลรังสี
หลักการเชื่อมต่อ: ใช้ Integration Layer (ESB/Adapter) เป็นตัวกลาง — ระบบ DIS ไม่เรียกฐานข้อมูลระบบอื่นโดยตรง · มี Message Queue กันงานหลุดเมื่อปลายทางล่ม · บันทึก Audit ทุกการแลกเปลี่ยนข้อมูล

5โซนความปลอดภัย (Security Zones)

ออกแบบตามหลัก Defense in Depth และ Zero Trust — ไม่เชื่อถือทราฟฟิกใดโดยปริยาย ตรวจสอบสิทธิ์ทุกชั้น แบ่งเครือข่ายเป็นโซน

INTERNET ──► [ WAF / DMZ Zone ]──► [ App Zone ]──► [ Data Zone ] LB, Reverse Proxy API, App Server DB, Storage (public-facing) (no public IP) (no internet) │ │ │ Firewall(1) Firewall(2) Firewall(3) Zero Trust: ทุกคำขอผ่าน Auth/SSO + RBAC + mTLS ระหว่างบริการ
โซนมาตรการ
DMZ (Public)WAF, DDoS protection, TLS termination, เปิดเฉพาะ 443
App Zoneไม่มี Public IP, mTLS ระหว่างบริการ, API Gateway บังคับ Auth
Data Zoneเข้าถึงจาก App Zone เท่านั้น, เข้ารหัสข้อมูล (at-rest), แยก network
Managementเข้าถึงผ่าน Bastion/VPN + MFA, แยก network จัดการ

6การขยายระบบและความพร้อมใช้สูง (Scalability & High Availability)

  • Horizontal Scaling — App Server แบบ stateless เพิ่ม instance หลัง Load Balancer ได้ตามโหลด
  • Database HA — Primary + Replica (อย่างน้อย 1), Automatic Failover, อ่านแยกไป Replica (read scaling)
  • Caching — Redis ลดภาระ DB สำหรับข้อมูล master/lookup
  • No Single Point of Failure — LB คู่, App หลาย instance, ลิงก์เครือข่ายสำรอง
  • เป้าหมายความพร้อมใช้ (SLA): 99.5% ต่อปี (เวลาหยุดทำงานสะสมไม่เกิน __ ชม./เดือน)

7แผนกู้คืนภัยพิบัติ (Disaster Recovery)

รายการรายละเอียด
RPO (Recovery Point Objective)≤ 15 นาที — ข้อมูลที่ยอมสูญเสียได้สูงสุด
RTO (Recovery Time Objective)≤ 4 ชั่วโมง — เวลากู้คืนระบบให้กลับมาใช้งาน
การจำลองข้อมูลDB Replication ไป DR Site แบบ async + Object Storage sync
การสำรองข้อมูลFull backup รายวัน + Incremental, เก็บนอกสถานที่ (offsite)
การทดสอบ DRซ้อม Failover อย่างน้อย ปีละ 1 ครั้ง พร้อมบันทึกผล
⚠️ ข้อควรระวัง: ข้อมูลผู้ป่วย (PII/ข้อมูลสุขภาพ) ต้องเข้ารหัสทั้ง at-rest และ in-transit ในทุกสภาพแวดล้อมรวมถึง DR Site และสื่อสำรองข้อมูล (สอดคล้อง PDPA)
