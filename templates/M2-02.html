1วัตถุประสงค์และวิธีอ่านตาราง

1.1 วัตถุประสงค์ — ทำให้ทุกความต้องการตาม TOR มีร่องรอย (traceability) ตลอดวงจรพัฒนา เพื่อใช้ตรวจรับว่า “สิ่งที่ส่งมอบตรงกับ TOR ครบ” และใช้วิเคราะห์ผลกระทบเมื่อมี change request

1.2 ความหมายของแต่ละคอลัมน์

คอลัมน์ความหมาย
Req IDรหัสความต้องการภายในโครงการ (สอดคล้องกับ Fit/Gap M2-01)
อ้าง TORข้ออ้างอิงใน TOR / ภาคผนวก ก
คำอธิบายความต้องการสาระของข้อกำหนดโดยย่อ
เอกสารออกแบบรหัสเอกสาร/หัวข้อออกแบบที่รองรับ (เช่น SDD-xx, Use Case)
โมดูล / Buildโมดูลหรือรายการพัฒนา/Config ที่ทำให้เกิดจริง
Test Case IDรหัสกรณีทดสอบที่พิสูจน์ความต้องการนี้
สถานะสถานะปัจจุบัน: ออกแบบ / พัฒนา / ทดสอบ / ผ่าน

1.3 รหัสสถานะที่ใช้: ยังไม่เริ่ม กำลังออกแบบ/พัฒนา รอ/อยู่ระหว่างทดสอบ ผ่านแล้ว

3สรุปความครอบคลุม (Coverage Summary)

นับจำนวนความต้องการที่มีร่องรอยครบในแต่ละขั้น เพื่อแสดงร้อยละความครอบคลุม

ขั้นตอนจำนวนที่ครอบคลุมจากทั้งหมดร้อยละ
มีเอกสารออกแบบ (Design coverage)______%
มีโมดูล/งานพัฒนา (Build coverage)______%
มี Test Case (Test coverage)______%
ทดสอบผ่านแล้ว (Verified)______%
เป้าหมาย: ก่อน UAT/ตรวจรับ ทุกบรรทัดต้องมี Design + Build + Test Case ครบ 100% และสถานะ “ผ่านแล้ว” = 100% ของข้อบังคับ (Must)
