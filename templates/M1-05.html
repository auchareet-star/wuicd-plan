1กลยุทธ์การพัฒนา (Development Strategy)

  • พัฒนาแบบ รายโมดูลเป็นเฟส (Phased / Iterative) ส่งมอบทีละชุดเพื่อให้ทดสอบและตรวจรับได้เร็ว
  • ยึดหลัก Config ก่อน Develop — ใช้ความสามารถมาตรฐานก่อน เขียนโค้ดเฉพาะส่วน Gap (ต่อยอดจาก Fit/Gap M2-01)
  • จัดลำดับตาม dependency และความสำคัญต่อ Go-Live (โมดูลแกน HIS/ทันตกรรมก่อน)
  • ใช้ระบบควบคุมเวอร์ชันและ CI/CD เพื่อ deploy ขึ้น Test/Staging อัตโนมัติ
หลักการ: โมดูลที่จัดเป็น CONFIG ทำก่อน, GAP ที่ซับซ้อนและพึ่ง integration วางไว้ให้สอดคล้องกับช่วงที่ทีม HIS เดิมว่าง

4นิยามความเสร็จสมบูรณ์ (Definition of Done)

  • พัฒนา/Config ครบตามความต้องการ (SRS) ที่ sign-off แล้ว
  • ผ่าน Unit Test / Code Review
  • ผ่าน Integration Test กับโมดูล/ระบบที่เกี่ยวข้อง
  • ไม่มี defect ระดับ Critical/High คงค้าง
  • มีเอกสารประกอบ (คู่มือ/หมายเหตุการตั้งค่า) ครบ
  • Deploy ขึ้น Staging และพร้อมให้ผู้ใช้ทดสอบ UAT
  • เจ้าของงานยืนยันยอมรับฟังก์ชัน (Acceptance)

5การเชื่อมต่อระบบ (Integration) และการใช้ Mock/Stub

การเชื่อม HAIT Plus, SAP, PACS, LIS/RIS, WU-HRMS พึ่งทีม HIS เดิมเป็นช่วง ๆ จึงต้องลดการพึ่งพาแบบ blocking ด้วย mock/stub

  • ออกแบบ Interface ตามสัญญา/สเปก (อ้าง M2-06 Interface Design) ก่อนทีมเดิมจะว่าง
  • สร้าง Mock/Stub ของ HAIT Plus เพื่อให้ Dev พัฒนา/ทดสอบต่อได้ระหว่างรอทีมเดิม
  • เมื่อทีม HIS เดิมว่าง (Consult Sprint) จึงทำ Integration Test กับระบบจริงและปรับแก้
  • บันทึกผลทดสอบ interface และ sign-off ร่วมกับทีมเดิม
⚠️ จุดวิกฤต: งาน Integration ที่พึ่งทีม HIS เดิมต้องจองคิวล่วงหน้าและทำเป็น batch — ใช้ mock/stub ครอบไว้เพื่อไม่ให้ Dev หยุดรอ
Dev ──▶ DIS ──▶ [ Mock/Stub HAIT Plus ] (ระหว่างรอทีมเดิม) │ Consult Sprint ───┴──▶ [ HAIT Plus จริง ] (Integration Test + sign-off)
